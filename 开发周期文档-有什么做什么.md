# 开发周期文档 — 「有什么做什么」微信小程序 MVP

---

## 总览

| 项目 | 说明 |
|------|------|
| 开发模式 | 单人开发 / 微信小程序原生框架 |
| 内容策略 | 方案 A+C 混合：先写死5道菜跑通流程 → 再用爬虫批量获取100道菜 → 导入数据库 |
| 预估总工期 | **15-22 个工作日**（约 3-4.5 周） |
| 核心思路 | **先让程序跑起来，再慢慢喂数据。不要等数据齐了才写代码。** |

---

## 内容从哪里来 —— 一句话策略

> **不用实时抓取。** 用 MediaCrawler 一次性批量爬菜谱，存进自己的数据库。小程序不依赖任何外部 API。

```
你电脑上跑一次爬虫 → 导出 JSON → 清洗 → 导入云数据库 → 小程序读自己的库
```

为什么不做实时搜索：
- 小红书/抖音没有开放 API
- 爬虫不能在微信小程序里运行
- 实时爬有封号风险、维护成本高
- 菜谱这类内容不像新闻——不需要"实时性"，100道菜够用很久

## 阶段零：写死 5 道菜，跑通全流程（1-2 天）★ 第一步就做这个

**目标**：不依赖任何数据库、任何后端，纯粹用写死在代码里的 5 道菜数据，把三个页面全部跑通。

> 这是最重要的阶段。数据和逻辑都在前端，你可以立刻在微信开发者工具里看到效果、截图、发给朋友看。

## 技术选型速查

| 项 | 选择 | 备注 |
|----|------|------|
| 框架 | 微信小程序原生 | MVP 最简单 |
| 数据存储 | 微信云开发 · 云数据库 | 免运维、小数据量免费额度够用 |
| 图片存储 | 微信云开发 · 云存储 | 同上 |
| 数据爬取 | MediaCrawler（一次性使用） | 跑在你自己电脑上，不部署 |
| 状态管理 | 小程序 globalData | 三个页面不需要引入 MobX/Redux |
| UI 库 | 不用库，自己写 | 页面少，引入库反而拖慢 |

## 风险提示

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 小红书/抖音外链被微信拦截 | 跳转失效 | 详情页保留文字步骤作为兜底；搜索关键词链接作为备选 |
| 爬虫跑不出足够数据 | 数据量不够 | 最低底线：手动补 30 道高频菜（一个下午能做完） |
| 云开发费用超限 | 无法继续服务 | MVP 阶段数据量极小，基本不会超 |
| 微信审核不通过 | 无法上线 | 提前确认类目资质；内容不涉及医疗/保健品 |
| 页面功能在真机上体验差 | 用户流失 | 阶段四必须做真机测试，不要只在模拟器里跑 |

---

## 三个关键建议

1. **不要等数据齐了再写代码。** 阶段零用 5 道菜先把全流程跑通，这是最快获得正反馈的方式。
2. **不要在数据源上纠结太久。** 爬虫能跑就跑，跑不出来就手动整理——30 道菜覆盖 80% 的搜索场景。
3. **先上线，再迭代。** 100 道菜 + 三个干净页面 + 能跳转 = 可以上线了。收藏、历史、推荐这些都是后面的事。
