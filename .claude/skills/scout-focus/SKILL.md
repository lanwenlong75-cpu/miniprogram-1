# Focus 方法 - 聚焦层侦察

你被 Leader 派遣并指示用 focus 方法。本 skill 描述具体执行方法。

## 与 Quick 模式的核心差异

| 维度 | Quick | Focus |
|------|-------|-------|
| 阅读单位 | 匹配片段（±20行） | **完整函数体 + 类头部** |
| 依赖追溯 | 不做 | **追溯 1 层** |
| 调用方分析 | 不做 | **grep 调用方位置** |
| 测试文件 | 跳过 | **扫描相关测试** |
| 架构分析 | 不做 | **画出局部依赖图** |
| 输出风险 | 不评估 | **给出影响面分析** |

## 工作步骤

### Step 1：定位与规模评估
1. 用 grep / glob 定位关键词
2. 列出 3-8 个最相关文件（focus 方法的合理范围）
3. 如果相关文件超过 8 个：
   - 不要自己降级到只看一部分
   - 反馈 Leader："文件规模超出 focus 合理范围，是否升级到 deep？"
   - 等待 Leader 决定

### Step 2：函数级阅读（focus 的核心）
对每个相关文件：

#### 必读内容
- 关键函数的**完整函数体**（不是片段）
- 类定义的**头部**：class 声明 + `__init__` + 公开方法签名
- 模块顶部的 import 区（理解依赖）
- 模块级常量、装饰器（可能影响行为）

#### 不读内容
- 与任务无关的其他函数实现细节
- 私有辅助函数的实现（除非被关键函数调用）
- 文档字符串的长篇说明（看摘要即可）

### Step 3：依赖追溯（1 层即止）

对每个关键函数做两件事：

#### 调用方追溯（"谁调用了我"）
1. grep 函数名 / 类名，找出所有调用位置
2. 记录调用方文件:函数（不读调用方实现，只记位置）
3. 如果调用方 ≥ 5 个 → 反馈 Leader 考虑升级 deep

#### 被调用方追溯（"我调用了谁"）
1. 看函数体里调用了哪些其他函数
2. 对关键调用读**签名**（不读实现）
3. 跨模块调用要标注（理解模块边界）

**关键约束：只追溯 1 层**。
- ✅ A 调用 B → 看 B 的签名
- ❌ B 调用 C → 不再追下去（那是 deep 的活）

### Step 4：测试文件扫描

1. grep 测试文件中相关函数/类的引用
2. 列出测试覆盖位置（不读测试实现细节）
3. 标记关键的测试用例命名（理解行为契约）

### Step 5：构建局部依赖图

在脑内整理：
- 哪些文件因为这次改动需要同步修改
- 哪些是直接影响（必改）
- 哪些是间接影响（可能要改）
- 哪些是契约边界（改了会破坏外部）

## 输出契约（Focus 模式）

```markdown
## Scout 报告（focus）：{任务一句话}

### 涉及文件清单（含职责）
- `path/to/file1.py` — 文件职责一句话
- `path/to/file2.py` — 文件职责一句话
- `tests/test_file1.py` — 相关测试覆盖范围

### 关键签名（含上下文）
- `path1:funcA(x: int, cb: Callable) -> Result`
  - 用途：...
  - 调用方：path2:method, path3:handler
  - 内部调用：path4:helper
  
- `path2:ClassB`
  - 职责：...
  - 关键方法：method1, method2
  - 继承自：BaseB（来自 path5）

### 局部依赖图
```
path1:funcA
  ├─ 被调用：path2:method, path3:handler
  ├─ 调用：path4:helper（同模块）
  └─ 调用：path5:utility（跨模块）

path2:ClassB
  └─ 实例化于：path6:factory

- **直接影响**（必改）：funcA, funcB

- **契约边界**（改动需谨慎）：path1:funcA 是公开 API，被外部模块依赖
- **测试覆盖**：tests/test_file1.py 有 3 个相关用例

- {如发现架构约束、命名规则、SDK 限制}
- {如发现警示注释、TODO/FIXME}
- {如发现明显的不一致或重复逻辑}

### 改动建议（基于 focus 观察，不是完整方案）
- 主要改动点：path1:funcA 的 X 部分
- 同步修改：path2:ClassB.method 签名（因为依赖关系）
- 测试更新：tests/test_file1.py 需要新增 case

- ❌ 未做 2 层以上的依赖追溯
- ❌ 未通读相关文件全文，仅读了关键函数
- ❌ 未深入分析跨服务/跨模块的契约影响
- 以下情况建议 Leader 考虑升级 deep：
  - {如果发现某些线索指向更深问题}
  - {如果某个调用链路看起来比预期复杂}

## Focus 模式禁止行为

- ❌ **不要做 2 层以上依赖追溯**——A→B→C→D 这种深度是 deep 的活
- ❌ **不要全文通读文件**——只读关键函数和类头部
- ❌ **不要写完整方案**——你只做侦察，方案是 Leader 的事
- ❌ **不要返回代码原文**（>3 行禁止）
- ❌ **不要修改任何文件**

## Focus 模式允许的反馈升级

执行中如果发现以下情况，**反馈 Leader（不自动升级）**：

| 发现 | 反馈话术 |
|------|---------|
| 调用方 ≥ 5 个 | "影响面较大，建议升级 deep" |
| 发现继承链/装饰器链 | "存在多层抽象，1 层追溯不够" |
| 看到陷阱注释 | "发现 # HACK 注释，建议深读" |
| 涉及公共 API 修改 | "改动会影响对外契约，建议 deep" |
| 发现重复逻辑 | "多处类似实现可能存在隐藏不一致" |

反馈位置：在输出契约的"我的盲区声明"末尾追加"建议升级"段。

## Focus 方法的真实定位

Focus 是侦察的"主力工"——大部分中等任务用 focus 就够了。它的目标是：

✅ 让 Leader 写方案时**有依赖图、有影响面、有契约视角**  
✅ Token 成本可控（不烧全文）  
✅ 仍能发现"浅看会漏的局部协作问题"

❌ 不追求"摸清整个架构"——那是 deep 的事  
❌ 不追求"零盲区"——focus 必然有盲区，盲区要诚实声明
