# dev-builder

## 概述
JavaScript 项目的结构化开发方法论。通过强制性的预检和开发中纪律，防止范围蔓延、架构漂移和编译失败。

---

## 阶段一：预检（此时不要写代码）

### 1.1 读 DISPATCH PACK，严格按指令加载资源

**第一步**：打开 `工作目录/任务分发/{任务名}_方案规划.md`，读取最顶部的 DISPATCH PACK 块。

按 `implementer_resources` 字段加载资源。

**禁止行为**：
- ❌ 不要自己判断任务类型
- ❌ 不要根据任务描述"猜"该不该加载 UI 设计资源
- ❌ DISPATCH PACK 没列的资源，一律不加载

如果 DISPATCH PACK 缺失或字段不完整，**立即停止，报告 Leader**。

### 1.2 阅读现有代码

在写新代码之前，检查同一领域的现有实现：
- UI 模式：现有页面如何管理状态？
- 数据模式：现有数据层如何暴露数据？
- 架构：是否已建立 Repository/Service 模式？
- 命名：遵循现有的文件命名、包结构和组件命名规范

### 1.3 确认约束

查阅 `技术栈与构建环境.md` 和 `项目背景与架构.md`：
- 基础库版本 >= 2.2.3，推荐 3.16.0
- 禁止使用的库或 API
- 无依赖注入，通过 getApp() 获取全局实例
- 迁移规则（如果涉及数据库实体变更）

## 阶段二：开发

### 2.1 范围锁定
- 只实现任务清单中列出的项目
- 如果注意到相关的 bug 或改进点，记录在任务报告中——不要修复
- 如果方案有歧义，向 Leader 寻求澄清——不要假设

### 2.2 关键决策点索引刷新

在以下时刻，**必须**使用 Read 工具重新读取相关文件：

| 决策场景 | 必须刷新的内容 |
|---------|--------------|
| 修改某个类/模块之前 | 重新读取该类当前完整内容 |
| 调用一个已有函数之前 | 重新读取该函数定义，确认参数和返回值 |
| 修改数据库实体之前 | 重新读取该实体 + 所有引用它的数据访问层 |
| 修改公共工具类/函数之前 | 使用 Grep 查找所有调用方并读取关键调用 |
| 使用一个不确定的 API 之前 | 重新读取官方文档或项目内的使用示例 |

### 2.3 架构合规

| 层级 | 规则 |
|-------|-------|
| **UI** | 页面使用 WXML + WXSS，组件化使用 Component 构造器 |
| **ViewModel/State** | 使用 Page/Component 的 data 和 setData 管理状态 |
| **Repository/Service** | 使用 wx.cloud 云开发数据库，app.globalData 做全局状态 |
| **数据访问** | 通过 wx.cloud.database() 访问云数据库 |
| **DI** | 无依赖注入，通过 getApp() 获取全局实例 |

### 2.4 强制分段规则

**情况 A：任务清单包含 4 项或以上任务**
- 每完成 2 项任务 → 强制产出中间任务报告
- 完成后 2 项时 → 执行 `git commit` + 向 Leader 汇报进度

**情况 B：单个子任务预计超过 30 分钟**
- 必须在任务开始前向 Leader 申请拆分为更小的子任务

**情况 C：连续开发超过 1 小时**
- 强制自检代码质量，向 Leader 汇报

1. **编译**：`N/A（微信开发者工具内编译）` — 必须零错误通过
2. **预览**：如果 UI 有变更，验证渲染无崩溃
3. **测试**：`N/A`
4. **方案核对**：重读任务清单 — 每一项都打钩了？
5. **Diff 审查**：`git diff` — 有超出方案范围的变更？如果有，回滚。
