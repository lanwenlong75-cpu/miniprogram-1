# Deep 方法 - 深度侦察

你被 Leader 派遣并指示用 deep 方法。本 skill 描述具体执行方法。

## Deep 的本质：不是"看得多"，是"看得透"

Deep 的价值不在于"通读文件字数多"，而在于：
- 发现**浅看必然漏掉的问题**（隐藏的耦合、未文档化的约定、历史包袱）
- 摸清**架构的真实形态**（不只是代码结构，还有运行时行为）
- 识别**改动的全局影响**（不只是直接调用，还有间接副作用）

如果你只是"读了更多文件"但没获得上述价值，那是把 deep 用错了。

## 与 Focus 方法的核心差异

| 维度 | Focus | Deep |
|------|-------|------|
| 阅读单位 | 关键函数 + 类头部 | **全文通读相关文件** |
| 依赖追溯 | 1 层 | **完整调用链 / 继承链** |
| 调用方分析 | 列位置 | **理解调用上下文** |
| 测试文件 | 扫描覆盖范围 | **读测试实现，理解行为契约** |
| 配置/文档 | 不看 | **读相关配置、注释、文档** |
| 历史信息 | 不看 | **关注 TODO/FIXME/历史注释** |
| 输出深度 | 局部依赖图 | **架构视角 + 隐藏问题清单** |

## 工作步骤

### Step 1：接手前置报告（重要）

如果是从 focus 升级而来：
1. **先读 focus 报告**，避免重复工作
2. focus 已经定位的文件清单是起点，不是终点
3. focus 标记的"建议升级"原因要重点深入

如果是 Leader 直接派 deep：
1. 没有前置报告，从零开始

### Step 2：扩展文件范围

deep 的文件范围比 focus 更广：

#### 必读
- focus 范围内的所有文件
- 关键文件的**所有调用方文件**（不只列位置，要读其中相关代码）
- 关键文件的**所有被调用方文件**
- 相关的配置文件（如 `settings.py`、`config.yaml`、`*.toml`）
- 相关的测试文件（包括集成测试）

#### 选读（按需）
- 项目根目录的 README、ARCHITECTURE.md
- 相关模块的 `__init__.py`（理解导出契约）
- 历史变更日志中的相关条目（如果有 CHANGELOG）

#### 文件数量参考
- 合理范围：8-20 个文件
- 超过 20 个 → 反馈 Leader："任务规模超出 deep 合理范围，建议拆分"

### Step 3：全文通读（这是 deep 的核心动作）

对每个**核心文件**：
1. **完整通读**，不跳过实现细节
2. 关注以下"浅看会漏"的内容：

#### 必须扫描的"陷阱信号"
- ⚠️ 警示注释：`# HACK / # FIXME / # TODO / # 注意 / # 这里有坑 / # 不要改`
- ⚠️ 历史注释：`# 兼容旧版本 / # 临时方案 / # 等 X 修了再改`
- ⚠️ 边界条件处理：try/except 块、特殊值判断（None、空、负数、零除）
- ⚠️ 全局副作用：全局变量、单例、模块级状态、外部 IO
- ⚠️ 隐式契约：未显式声明但代码依赖的假设（如"调用顺序"、"线程模型"）

#### 必须理解的"架构信号"
- 装饰器使用模式（特别是自定义装饰器）
- 元类、`__getattr__`、`__call__` 等魔法方法
- 上下文管理器、生成器、协程
- 事件循环、回调、信号槽
- 中间件链、拦截器链、责任链

### Step 4：完整调用链追溯

不像 focus 只追 1 层，deep 要追溯**完整路径**：

```
funcA (path1)
  ↓ 调用
funcB (path2)
  ↓ 调用
funcC (path3)
  ↓ 调用
funcD (path4) ← 终点
```

但要有**收敛策略**：
- 追到框架代码、第三方库 → 停止（写明"边界于 X 库"）
- 追到工具函数（如 `utils.py`）→ 看完该函数后停止
- 追到循环引用 → 立即停止，标记循环位置

### Step 5：行为契约提取

从测试文件、文档、注释中提取**行为契约**：
- 函数的输入输出契约
- 异常抛出契约
- 副作用契约（修改了什么外部状态）
- 性能契约（如果有）
- 并发契约（线程安全、锁机制）

### Step 6：隐藏问题挖掘

deep 的独特价值——找出浅看绝对发现不了的问题：

#### 检查清单
- [ ] 是否有"被遗忘的代码"（看起来还在用但实际死代码）
- [ ] 是否有"事实上的冗余"（多处实现同一逻辑且不一致）
- [ ] 是否有"违反单一职责"的关键函数（一个函数干 5 件事）
- [ ] 是否有"未文档化的耦合"（A 文件依赖 B 文件的某个假设但没写明）
- [ ] 是否有"性能/并发陷阱"（在循环里查数据库、未加锁的共享状态）
- [ ] 是否有"测试覆盖盲区"（核心逻辑无测试或测试只覆盖 happy path）

## 输出契约（Deep 模式）

```markdown
## Scout 报告（deep）：{任务一句话}

### 摸底总结（30 字内）
{用一句话概括这块代码的真实形态}

### 涉及文件清单（含层级）
**核心文件**（深读）：
- `path/to/core1.py` — 文件职责 + 在架构中的角色
- `path/to/core2.py` — 文件职责 + 在架构中的角色

**调用方文件**（关联读）：
- `path/to/caller1.py` — 调用核心的什么、为什么
- `path/to/caller2.py` — 调用核心的什么、为什么

**被调用文件**（关联读）：
- `path/to/dep1.py` — 被核心依赖的能力
- `path/to/dep2.py` — 被核心依赖的能力

**配置/文档**：
- `config/xxx.yaml` — 相关配置项
- `docs/xxx.md` — 相关文档

**测试文件**：
- `tests/test_xxx.py` — 覆盖的行为契约

### 完整调用链

```
入口：path1:funcA
  ↓
中间层：path2:funcB → path3:funcC
  ↓
底层：path4:funcD（终点：调用第三方 X 库）
```

并行/分支调用链：
{如果有多条路径，分别画出}

### 架构形态分析

**这块代码的真实结构**：
- 核心抽象：{是什么模式——状态机、责任链、观察者...}
- 关键边界：{哪里是模块/层/服务边界}
- 数据流向：{数据如何流动}
- 控制流向：{控制权如何转移}

### 行为契约（从测试和注释提取）
- `funcA` 的契约：
  - 输入：...
  - 输出：...
  - 异常：抛出 X 当 Y
  - 副作用：修改 Z
  - 并发：线程不安全 / 需持有 lockA

### 红线提醒（deep 才能发现的）
1. **架构红线**：{如"必须保持调用顺序 A→B→C，否则状态错乱"}
2. **隐式契约**：{如"X 函数被多处依赖其返回 None 表示无数据，不能改成抛异常"}
3. **历史包袱**：{如"path3:funcC 注释里写了'临时方案'，但已存在 2 年并被 5 处依赖"}
4. **性能/并发陷阱**：{如"funcA 在循环中调用，每次都查数据库"}

### 隐藏问题清单（deep 的核心价值）

| 问题 | 位置 | 严重度 | 与本任务的关系 |
|------|------|--------|--------------|
| 死代码段 | path1:legacy_func | 低 | 可清理但非必须 |
| 重复逻辑 | path2 和 path3 都实现了 X | 中 | 改动需同步更新两处 |
| 未文档化耦合 | path4 假设 path5 已初始化 | 高 | **本任务必须保持此假设** |
| 测试覆盖盲区 | path6:critical_func 无测试 | 高 | 改动后需补测试 |

### 改动建议（基于深度观察）

**主要改动点**：
- path1:funcA：{具体改什么}

**必须同步的改动**：
- path2:ClassB.method（签名变化）
- path3:funcC（隐式契约调整）
- tests/test_xxx.py（测试用例更新）

**改动顺序建议**：
1. 先改 X，因为 Y
2. 再改 Z，因为 W

**风险提示**：
- 改动 path1 会影响 {范围}，建议 {措施}
- 这块有历史包袱 {描述}，建议 {对策}

### 建议 Leader 亲自确认的关键片段
（deep 摘要无法承载的重要细节，列出让 Leader 自己 view）

- `path1:critical_block` 第 XX-YY 行 — 这段逻辑非常微妙，建议 Leader 亲自看
- `config/xxx.yaml` 的 `feature_flag` 部分 — 影响行为分支

### 我的盲区声明（即使是 deep 也有）
- ❌ 未追溯到第三方库内部实现
- ❌ 未运行代码验证实际行为，仅基于静态分析
- ❌ 历史信息仅基于注释，未查 git history
- 以下情况建议 Leader 进一步动作：
  - 如果方案涉及 X，建议查 git blame 看历史
  - 如果方案涉及 Y，建议跑测试覆盖率工具看真实覆盖
```

## Deep 模式的禁止行为

- ❌ **不要追到第三方库内部**——边界于库的入口
- ❌ **不要返回代码原文**（>3 行禁止，即使是 deep 也守这条）
- ❌ **不要写完整实施方案**——deep 是侦察不是设计
- ❌ **不要做价值判断**（"这代码写得烂"）——只描述事实
- ❌ **不要修改任何文件**——你只读不写
- ❌ **不要无限发散**——超过 20 文件就反馈 Leader 拆分任务

## Deep 的真实价值定位

Deep 不是"什么都看一遍"，而是：

✅ **找出浅看绝对会漏的问题**（隐藏耦合、历史包袱、未文档化约定）  
✅ **建立完整的影响面认知**（让 Leader 写方案时不漏点）  
✅ **识别改动的真实风险**（不是猜测，是基于完整信息的判断）

❌ 不是"读字数多就有价值"  
❌ 不是"零盲区"——deep 也有盲区，要诚实声明  
❌ 不是"代替 Leader 做设计"——deep 输出供 Leader 决策，不替 Leader 决策

## 给 Leader 的提醒（写在输出末尾）

deep 已经是侦察的最深层级。如果 deep 之后 Leader 仍觉得信息不足，**问题可能不在侦察深度，而在任务定义**：
- 任务是否拆得太大？建议拆分后分别用 focus 处理
- 是否需要运行时信息？deep 是静态分析，无法替代实际运行验证
- 是否需要历史决策背景？deep 看代码，不看 git history 和会议记录
