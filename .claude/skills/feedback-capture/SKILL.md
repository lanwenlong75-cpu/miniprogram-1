name: feedback-capture
description: 在需求确认完、写方案规划之前，检测刚才几轮对话中是否包含用户反馈信号，有则提炼并写入 feedback 文件。Leader 自己执行，不调子 agent。
---

# 触发时机
固定卡位：brainstorm 结束、用户批准方案后、task-dispatcher 启动前。
其他时机一律不主动加载。

# 执行步骤

## 第 1 步 · 关键词扫描
回看本次对话最近 5-10 轮，检测以下信号：
- "不对" / "你又忘了" / "记住" / "重新做"
- "下次别这样" / "应该是 X 不是 Y"
- 用户明显的情绪纠正（从中性变急促/不耐烦）
- 用户主动给出的规则（"以后都这样做"）

**没检测到 → 输出一行：`[feedback-capture] 无需记录` → 立即进入 task-dispatcher，不解释。**

## 第 2 步 · 反馈类型分类（检测到才执行）
- **structural**：长期有效的规则（命名规范、架构约束、流程要求）
- **contingent**：权宜之计（因当前能力/工具限制的妥协）

## 第 3 步 · 写文件
路径：`工作目录/feedback/{领域}-{问题简述}.md`

**已存在同名文件**：打开，`count` 字段 +1，追加本次触发的简述（一行）。
**不存在**：按下方模板新建，并在 `工作目录/feedback/FEEDBACK-INDEX.md` 登记一行。

### 文件模板（写死，不要发挥）
```
# {问题简述}

- 类型：structural | contingent
- 领域：{UI / 架构 / 命名 / 流程 / ...}
- count：1
- 首次触发：{YYYY-MM-DD}
- 最近触发：{YYYY-MM-DD}

## 用户原话
> {直接引用用户那句话，不改写}

## 上下文
{2-3 句话说明：当时在做什么任务，触发了什么动作}

## 规则提炼
{一句话，祈使句，告诉未来的 AI 该怎么做}

## 触发记录
- {YYYY-MM-DD} {一行简述}
```

## 第 4 步 · 收尾
输出一行：`[feedback-capture] 已记录：{文件名}` → 立即进入 task-dispatcher。
**不要解释、不要邀功、不要问用户确认。**

# 静默原则
本 skill 全程不与用户互动。Leader 加载后自己跑完，只在主对话留一行状态（已记录 / 无需记录），其余动作在文件系统完成。
