# 多模块任务执行指南

> 当任务被判定为"需要拆分成多模块"时，按本指南执行以开启任务。

---
## 步骤1：方案规划撰写
1. 创建目录，存放多个子模块产出的方案规划、报告
**目录结构**
```
工作目录/
  任务分发/{功能名}/
    {功能名}_总纲.md
    {子模块1}_方案规划.md
    {子模块2}_方案规划.md
    ...
  任务执行/{功能名}/
    {子模块1}_开发报告.md
    {子模块2}_开发报告.md
    ...
  审查/{功能名}/
    {子模块1}_审查报告.md
    {子模块2}_审查报告.md
    ...
```

### **编写总纲**

```markdown
# {功能名}_总纲

- **任务类型**：`功能开发` | `UI 设计-优化` | `混合（先功能后优化）`
- **分支**：feat/xxx
- **子模块数量**：N 个

## 子模块清单

| 序号 | 子模块名 | 核心目标 | 前置依赖 |
|------|---------|---------|---------|
| 1 | {子模块1} | 一句话目标 | 无 |
| 2 | {子模块2} | 一句话目标 | {子模块1} |

## 全局约束

- 所有子模块共用分支：feat/xxx
- 红线：不能修改的文件 / 不能引入的依赖
- 数据契约：跨子模块共享的数据结构定义
```

### 编写 子模块方案规划

每个子模块一份独立方案规划，必填"当前模块"字段：

```markdown
- **当前模块**：
  - 所属功能：{功能名}
  - 子模块：{子模块名}（第 N / 共 M 个）
  - 前置依赖：{前一个子模块名，首模块填"无"}
  - 后置消费方：{后一个子模块名，末模块填"无"}
```

**第一层：契约**
- 验收标准：什么状态算"做完"
- 输入输出定义：输入、输出、影响哪些数据
- 红线约束：不能碰哪些文件、不能用哪些 API

**第二层：任务清单**
- 不可再拆的最小交付单元
- 每个任务有一行可验证的完成标准
- 标明依赖关系

**第三层：任务追加**
- 因为返工，而新增的要优化的需求
- 没有就留空



**子模块拆分原则**：
- 每个子模块必须可独立验收（代码正确、不破坏编译）
- 依赖单向，禁止循环依赖
- 风险高 / 不确定性强的模块放前面

---

## 步骤2：锁定基准

总纲和子模块方案规划编写完成后，立即 commit：
```bash
git add "工作目录/任务分发/{功能名}/"
git commit -m "plan: {功能名}总纲与子模块方案规划"
```

---

## 步骤3：代码编写

**核心规则**：
- 每个子模块必须调用独立的 implementer session
- 严禁把多个子模块塞进一个 session
- 子模块之间通过"开发报告"传递上下文

**第一个子模块**

**Leader Prompt**：
```
【总纲】工作目录/任务分发/{功能名}/{功能名}_总纲.md
【当前子模块方案】工作目录/任务分发/{功能名}/{子模块1}_方案规划.md

请读取以上文件，只完成当前子模块的方案内容。
完成后产出开发报告到：工作目录/任务执行/{功能名}/{子模块1}_开发报告.md
```

**后续子模块**

**Leader Prompt**：
```
【总纲】工作目录/任务分发/{功能名}/{功能名}_总纲.md
【前一个子模块开发报告】工作目录/任务执行/{功能名}/{子模块N-1}_开发报告.md
【当前子模块方案】工作目录/任务分发/{功能名}/{子模块N}_方案规划.md

请读取以上文件，基于前面已完成的工作，继续完成当前子模块。
完成后产出开发报告到：工作目录/任务执行/{功能名}/{子模块N}_开发报告.md
```

---

## 步骤5：代码审查

**核心规则**：
- 每个子模块必须调用独立的 code-reviewer session
- 严禁把多个子模块塞进一个 session
- 子模块之间通过"开发报告"传递上下文

**Leader Prompt**：
```
【总纲】工作目录/任务分发/{功能名}/{功能名}_总纲.md
【当前子模块方案】工作目录/任务分发/{功能名}/{子模块N}_方案规划.md
【当前子模块开发报告】工作目录/任务执行/{功能名}/{子模块N}_开发报告.md
【前一个子模块审查报告】工作目录/审查/{功能名}/{子模块N-1}_审查报告.md -首个子模块不需要

请读取以上文件，审查当前子模块的实现是否符合方案规划。
```

---

## 步骤6：审查闭环

- code-reviewer 没发现严重问题、符合方案规划 → 进入下一个子模块或下一步骤
- code-reviewer 发现问题 → Leader 调用 implementer（同一个子模块）进行修复，修复后必须重新审查，直到通过

---

## 步骤7：用户审核与合并主干

全部子模块完成后，向用户展示整体执行摘要并询问：

```
当前任务已完成，改动概要：
- xxx
- xxx

请问对此次修改是否有任何问题？
如果没有，请告诉我"没有"，我会合并到主干。
```

- 用户说"有" → 按 `REWORK_GUIDE.md` 执行（只涉及某个子模块则重做该子模块；涉及全局设计则重新评估总纲）
- 用户说"没有" → 合并分支：

```bash
git checkout master
git merge feat/xxx --no-ff
git branch -d feat/xxx
```
