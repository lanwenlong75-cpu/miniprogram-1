# 步骤1
## 任务复杂度初判（在派 scout 之前做）

根据用户需求的语义特征做初判：

### 无需让scout额外调用Skill的需求特征
- 改动范围明确且小：单个按钮/字段/接口/函数
- 动词是：改、调、修、改文案、改样式、改默认值
- 目标对象是叶子节点：UI 组件、配置项、错误信息
- 不涉及数据流、状态、协议变更

例子：
- "改一下提交按钮的颜色"
- "用户名最大长度从 20 改到 30"
- "修复这个报错的文案"

### 需要让scout调用 scout-focus Skill的需求特征
- 改动跨 2-4 个文件可预见
- 动词是：加、扩展、新增、集成、对接
- 涉及一个完整功能但范围有界
- 可能改契约但只在局部

例子：
- "用户表加一个 phone 字段"
- "登录页加一个忘记密码入口"
- "对接一下短信验证码服务"

### 需要让scout调用 scout-deep Skill的需求特征
- 动词是：重构、迁移、重写、统一、规范化
- 涉及核心抽象、状态机、数据模型
- 影响公共 API 或对外契约
- 跨多个模块/服务的横切问题

例子：
- "把支付流程改成状态机驱动"
- "统一所有服务的错误处理"
- "把单体拆成微服务"

### 判断不出来时的默认值
- 如果你判不准 → 默认让 scout 调用 `scout-focus` Skill，
  （宁可多花一点 token，也比漏看好）

# 步骤2
## 反直觉检查清单（初判后强制做一次）

初判完成后，**问自己以下 5 个问题**，任一回答"是"就升一级：

1. 用户用的是简单动词（"加""改"），但对象是核心概念（"缓存""队列""认证""状态"）吗？
   → 是 → 指派 scout-deep Skill

2. 需求描述少于 15 字，但隐含跨模块影响吗？
   → "加个缓存""改成异步""做个限流" → 是
   → 指派 scout-deep Skill

3. 需求里有"统一""所有""全部""整个"这类全称词吗？
   → 是 → 指派 scout-deep Skill

4. 需求涉及"性能""安全""并发""数据一致性"这类系统性属性吗？
   → 是 → 至少 scout-focus，倾向 scout-deep Skill

5. 项目是你（Leader）首次接触/不熟悉的吗？
   → 是 → 至少 scout-focus Skill

## 反向降级（也要做一次）

1. 需求里明确说"只改 X 文件"或"在 Y 函数里"吗？
   → 是 → 可以无需额外调用skill
2. 需求是修一个明确的 bug 且复现路径清晰吗？
   → 是 → 可以无需额外调用skill

# 传递格式
输入给 scout：
- scout需要调用的 额外skill: 无/scout-focus/scout-deep
- 任务一句话描述
- 涉及的领域 / 模块 / 关键词

接收 scout 报告。**Leader 自己不要 Read 任何源码文件。**

如果 scout 报告中有"建议 Leader 亲自确认"的片段，Leader 才被允许 Read 那个具体文件的具体位置（只读那一段，不读全文）。
