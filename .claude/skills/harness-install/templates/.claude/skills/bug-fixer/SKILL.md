# bug-fixer

## 概述
系统化四阶段调试方法论。防止掩盖症状或造成回退的猜测式修复。核心原则：**没有证据就不修复**。

## 何时使用
- `{{BUILD_CMD}}` 失败
- 应用运行时崩溃
- 单元测试或集成测试失败
- 行为与方案不符（如 UI 不更新、数据不持久化）
- 代码编译但结果错误

## 何时不使用
- 修复显而易见且已验证（如 IDE 捕获的缺失 `import`）
- 正在添加新功能（改用 `dev-builder`）

---

## 阶段一：收集证据

**此时不要碰代码。**

### 1.1 捕获错误
- 完整堆栈跟踪（不只是第一行）
- 错误信息原文（逐字复制，不要转述）
- 跟踪中的文件路径和行号

### 1.2 可靠复现
- 能用相同步骤连续触发 3 次？
- 全新安装是否也会发生？
- 在 debug 和 release 构建中都会发生？

### 1.3 隔离触发点
- 损坏前最后的变更是什么？(`git log --oneline -5`)
- 回退最后一个变更能修复吗？
- 是否特定于设备/环境？

**红旗**：如果不能可靠复现，说明你还没理解这个 bug。

---

## 阶段二：分析模式

### 2.1 分类错误

{{BUG_CATEGORIES}}

### 2.2 检查反馈池
阅读 `工作目录/feedback/` 中类似的过往错误

### 2.3 形成假设
准确陈述**一个**假设。

**红旗**：如果假设中包含"可能"或"大概"，回到阶段一。

---

## 阶段三：验证假设

### 3.1 最小变更
只改变**一件事**来测试假设。不要重构。

### 3.2 编译并运行
- `{{BUILD_CMD}}` — 必须通过
- 复现原始场景

### 3.3 假设错误？
- 回退变更
- 形成**新的**单一假设
- 重复

---

## 阶段四：实施修复 + 回归测试

### 4.1 编写修复
基于已验证的假设，实施最小正确修复。

### 4.2 验证修复
- `{{BUILD_CMD}}` 通过
- 原始错误不再复现
- `{{TEST_CMD}}` 通过

### 4.3 回归检查
- 运行完整功能流程
- 检查可能共享相同模式的相关页面/功能

---

## 常见借口（停）

| 借口 | 现实 |
|----------------|---------|
| "我先试试这个看看" | 你在猜测。回到阶段二。 |
| "大概是个线程问题" | "大概"不是假设。证明它。 |
| "修复很明显" | 如果真明显，就不需要这个技能了。验证。 |
| "顺便清理一下" | 不。一次只改一件事。 |
