# PRD — 「有什么做什么」微信小程序

---

## 一、产品概述

| 维度 | 说明 |
|------|------|
| 产品名称 | 有什么做什么 |
| 产品定位 | 输入冰箱里的食材，秒出能做的菜谱 |
| 平台 | 微信小程序 |
| 目标用户（MVP） | 饿了不想出门买菜的上班族 |
| 产品类型 | 效率工具（MVP 阶段不讲留存，先讲有用） |

### 1.1 用户故事（核心）

> 我下班到家、打开冰箱，看到鸡蛋、番茄、挂面。不想出门买菜，也不想对着小红书/抖音刷 20 分钟还不知道做什么。我打开「有什么做什么」，勾选"鸡蛋 + 番茄 + 挂面"，5 秒内看到 15 道能做的菜，点开一个番茄鸡蛋面，跟着链接去小红书看视频——1 分钟搞定，开火做饭。

### 1.2 核心价值

**帮用户在 1-2 分钟内从"我有什么"到"我能做什么"。**

对比现状：
- 小红书/抖音：需要自己搜、自己筛、漫无目的刷 → 时间长、决策累
- 本产品：输入食材 → 直接给结果 → 精准匹配 → 省时间、省决策

## 二、MVP 功能范围

### 2.1 要做的事（Must Have）

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 食材选择 | 高频食材标签（可点选）+ 自定义文本输入 | P0 |
| 菜谱匹配 | 根据已选食材匹配菜谱，按匹配度排序 | P0 |
| 结果卡片列表 | 菜品名称 + 封面图 + 所需食材 + 匹配百分比 | P0 |
| 菜谱详情页 | 食材清单 + 简要步骤 + 外部跳转链接（小红书/抖音） | P0 |
| 预置菜谱数据库 | MVP 先人工整理 60-100 道常见家常菜 | P0 |
| 匹配度计算 | 已选食材命中菜谱食材的比例 | P0 |

### 2.2 暂时不做的事（Not in MVP）

| 功能 | 原因 |
|------|------|
| 拍照识别食材 | 技术成本高，先验证核心流程 |
| 语音输入 | 同上，先用标签+文字验证需求 |
| 用户收藏/历史记录 | 工具属性，先验证"找到就走"的模式能否跑通 |
| 用户上传食谱 | 冷启动阶段无内容，先走预置路线 |
| 实时抓取小红书/抖音 | 无开放 API，爬虫不合规，MVP 用预置数据+手动外链替代 |
| 缺食材推荐（"再加酱油就能做XXX"） | V1 优化项，MVP 先告知"未找到" |
| 食材购物清单 | 超出 MVP 范围 |

## 四、数据结构

### 4.1 菜谱

```json
{
  "id": "recipe_001",
  "name": "番茄炒蛋",
  "coverImage": "cloud://xxx/cover/tomato-egg.jpg",
  "ingredients": [ // 必选食材
    { "name": "鸡蛋", "amount": "2个" },
    { "name": "番茄", "amount": "2个" }
  ],
  "optionalIngredients": [ // 可选食材（缺了也能做）
    { "name": "葱", "amount": "少许" },
    { "name": "糖", "amount": "少许" }
  ],
  "steps": [
    "鸡蛋打散，番茄切块",
    "热油炒蛋，盛出备用",
    "炒番茄出汁，加蛋翻炒",
    "加盐调味出锅"
  ],
  "links": {
    "xiaohongshu": "https://www.xiaohongshu.com/explore/xxxxx",
    "douyin": "https://v.douyin.com/xxxxx/"
  },
  "category": "家常菜",
  "cookTime": "10分钟",
  "difficulty": "简单"
}
```

### 4.2 食材标签

```json
{
  "id": "ingredient_001",
  "name": "鸡蛋",
  "displayName": "鸡蛋",
  "icon": "🥚",
  "isHighFrequency": true,  // true = 在首页展示
  "sortOrder": 1
}
```

### 4.3 匹配度算法

```
匹配度 = 已选食材 ∩ 必选食材 的数量 / 必选食材总数 × 100%

示例：
  用户选了 [鸡蛋, 番茄]
  番茄炒蛋必选 [鸡蛋, 番茄]  → 2/2 = 100%
  番茄蛋汤必选 [鸡蛋, 番茄, 葱]  → 2/3 = 67%

排序优先级：匹配度从高到低
```

## 五、UI/UX 要求

| 要求 | 说明 |
|------|------|
| 视觉风格 | 简洁、暖色调、有食欲感（橙色/暖黄系） |
| 首页原则 | 90% 场景靠标签点选完成，少打字 |
| 搜索结果 | 匹配度 100% 的卡片特殊标识（绿色勾），缺食材的标注缺了几样 |
| 详情页 | 食材清单 + 文字步骤一目了然，外链按钮醒目 |
| 空状态 | 未找到结果时友好提示，建议减少或更换食材 |
| 加载速度 | 搜索结果本地匹配（预置数据量小），不依赖网络请求 |

## 六、内容策略（MVP）

### 6.1 菜谱来源

- MVP 阶段：人工整理 60-100 道常见家常菜
- 整理内容：菜名 + 封面图（可使用无版权图片）+ 食材清单 + 简要步骤 + 小红书/抖音对应教程链接
- 覆盖场景：一人食、家常快手菜、冰箱剩菜组合

### 6.2 整理优先级

优先覆盖**最常被搜索的食材组合**：
- 鸡蛋 + 番茄
- 鸡蛋 + 葱花
- 土豆 + 青椒
- 猪肉 + 青椒
- 豆腐 + 葱花
- 鸡胸肉 + 西兰花
- 黄瓜 + 鸡蛋
- 各种蔬菜 + 蒜蓉

## 七、成功指标（MVP 阶段）

MVP 阶段不以日活/留存为核心，先以**功能完整性**为衡量标准：

| 指标 | 目标 | 衡量方式 |
|------|------|----------|
| 菜谱覆盖率 | 覆盖 10 种以上常见食材的所有组合 | 人工检查 |
| 搜索命中率 | 常见食材组合至少返回 3 条以上结果 | 人工测试 |
| 核心路径可用性 | 选食材 → 看结果 → 点开详情 → 跳转外链，全流程无阻断 | 真机走查 |
| 小程序审核通过 | 通过微信审核上线 | 提交审核 |

## 八、V1 规划（MVP 验证后的迭代方向）

| 优先级 | 功能 | 触发条件 |
|--------|------|----------|
| P1 | 缺食材推荐（"再加酱油就能做 XXX"） | MVP 验证核心路径可用 |
| P1 | 收藏功能 | 用户反馈有复用需求 |
| P2 | 历史搜索记录 | 同上 |
| P2 | 拍照识别食材 | 标签输入无法覆盖长尾场景 |
| P3 | 用户上传食谱 | 预置内容量不满足需求 |
| P3 | 语音输入 | 厨房场景不方便打字 |
